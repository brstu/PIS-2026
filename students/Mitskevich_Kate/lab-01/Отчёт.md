### 2. Use-case описание

👉 **Ссылка на файл:** https://github.com/Katapo13/PIS-2026/use-case.md

**Основной сценарий:** Добавление записи о доходе
**Первичный актор:** Пользователь

**Цель:** Зафиксировать поступление денежек и автоматически их распределить по категориям

**Краткое описание основного потока:**
1. Пользователь выбирает «Добавить доход»
2. Пользователь вводит сумму и подтверждает операцию
3. Система проверяет настройки бюджета (процентное распределение)
4. Система рассчитывает доли для каждой категории
5. Система создает запись о транзакции и обновляет балансы категорий
6. Система подтверждает успешное сохранение.

**Альтернативные потоки:**
1. A1: Категории не настроены — вся сумма зачисляется в категорию «Нераспределенное».
2. A2: Пользователь редактирует сумму перед финальным сохранением

**Исключительные ситуации:** 
1. E1: Введена отрицательная или нулевая сумма — система выдает ошибку валидации
2. E2: Сбой базы данных при обновлении балансов — транзакция откатывается

### 3. Диаграммы последовательности (Sequence Diagrams)

#### 3.1. Happy Path (успешный сценарий)

**Описание потока:**
- Пользователь инициирует добавление дохода
- Система (Transaction Service) в рамках одной транзакции создает запись о поступлении и обновляет остатки в таблице категорий (Category Service).

#### 3.2. Error Case (сценарий с ошибкой)

👉 **PlantUML исходник:** [sequence-error-payment.puml]([diagrams](https://github.com/Katapo13/PIS-2026/diagrams/)/sequence-error-payment.puml)

**Описание потока:**
- При попытке обновить баланс категории происходит сбой базы данных
- Система выполняет ROLLBACK, чтобы сумма дохода не «повисла» в воздухе без фактического зачисления на категории.

### 4. Gherkin-сценарии

👉 **Ссылка на файл:** [scenarios.feature](https://github.com/Katapo13/PIS-2026/scenarios.feature)

**Реализовано сценариев:** 4

**Список сценариев:**
1. ✅ **Успешный сценарий** Добавление дохода 1000$ при распределении 50/50.
2. ✅ **Ошибка:** Попытка добавить расход, превышающий текущий баланс категории.
3. ✅ **Ошибка:** Ввод текста вместо суммы в поле дохода.
4. ✅ **Ошибка:** Сохранение категории с суммарным процентом > 100%.

**Пример сценария:**
```gherkin
Feature: Добавление дохода

Scenario: Успешное добавление дохода
  Given в системе настроены категории "Еда" (40%) и "Сбережения" (60%)
  When пользователь добавляет доход "100" рублей
  Then баланс категории "Еда" увеличивается на 40 рублей
  And баланс категории "Сбережения" увеличивается на 60 рублей
```

### 5. Анализ границ ответственности

👉 **Ссылка на файл:** [analysis.md](https://github.com/Katapo13/PIS-2026/analysis.md)

#### 5.1. Транзакционные границы

| Операция | Синхронная/Асинхронная | Откат при ошибке | Retry-стратегия | Идемпотентность |
|----------|------------------------|------------------|-----------------|-----------------|
| Валидация суммы | Синхронная | Нет | Нет | Да |
| Создание записи транзакции | Синхронная | Да | Нет | Да |
| Обновление балансов категорий | Синхронная | Да | Нет | Да |
| Генепация отчёта для графиков | Асинхронная | Нет | 3 попытки | Да |

#### 5.2. Обработка исключительных ситуаций

**Реализовано стратегий обработки:** 2

**Примеры:**

##### Исключительная ситуация 1: Сумма расхода

- **Условие возникновения:** Сумма расхода больше, чем есть в категории.
- **Обнаружение:** Проверка бизнес-правил перед выполнением UPDATE в БД.
- **Реакция:**  Прерывание операции.
- **Компенсация:** Не надо, изменений не было.
- **Уведомление пользователя:** Уведомление "Недостаточно средств в данной категории, экономить надо вообще-то"

##### Исключительная ситуация 2: Сумма процентов

- **Условие возникновения:** В настройках категорий сумма процентных ставок >100%
- **Обнаружение:** Когда пользователь захочет сохранить настройки, проверит условие
- **Реакция:** Не даёт сохранить настройки
- **Компенсация:** Не надо, изменений не было
- **Уведомление пользователя:** Уведомление "Распределение превысило 100%, это ты конечно молодец, но меня не обманишь, давай исправляй"

## Контрольные вопросы

**Подготовка к защите:**

1. Что такое транзакционная граница? Где она проходит в вашем сценарии?
   - Это предел, внутри которого группа операций выполняется как единое целое (ACID). В моем случае — это одновременное создание записи о доходе и обновление балансов всех связанных категорий.

2. Почему операция X выбрана синхронной, а Y - асинхронной?
   - Потому что расчет сложной аналитики может занимать время и не должен блокировать процесс сохранения денег пользователем, вдруг ему прям очень срочно нужно.

3. Как обеспечить идемпотентность при повторных запросах?
   - Для обеспечения идемпотентности (чтобы при случайном двойном клике доход не зачислился дважды) я буду использовать Idempotency Key. Клиентское приложение генерирует уникальный UUID для каждой новой транзакции и отправляет его в заголовке запроса. Сервер сохраняет этот ключ в базе данных. Если приходит запрос с уже существующим ключом, сервер не создает новую запись, а просто возвращает успешный ответ (HTTP 200) на основе уже выполненной ранее операции.

4. Что произойдёт, если внешний сервис вернёт ошибку после частичного выполнения операции?
   - Если под «внешним сервисом» подразумевается сервис аналитики или уведомлений, то основная финансовая операция (запись в БД) останется выполненной, так как она критична.

5. Как система обнаружит, что внешний сервис недоступен?
   - Если система не будет овечать долгое время, если придёт код ответов или вообще пачка ошибок.

6. Какие данные нужно логировать для диагностики сбоев?
   - ID пользователя
   - Его транзакции
   - Категории
   - статус ошибок
   - время возникновения ошибки
