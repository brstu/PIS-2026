# Use-case 1: Настройка категорий бюджета

**Предметная область:** Финучёт «Где мои деньги» 💸

**Первичный актор:** Пользователь

**Вторичные акторы:**
- Finance API (Бэкенд-сервис)
- Database (Хранилище транзакций и категорий)

**Цель:**  Определить правила распределения будущих доходов.

---

## Предусловия

- Пользователь авторизован
- База данных доступна

---

## Основной поток (Happy Path)

1. Пользователь переходит в раздел «Настройки бюджета».
2. Система отображает список текущих категорий (если есть) и общую сумму процентов
3. Пользователь нажимает «Добавить категорию»
4. Пользователь вводит название (например, «Покушать») и процент (например, «60%»)
5. Пользователь нажимает «Сохранить».
6. Система проверяет:
   - Название не пустое
   - Процент — положительное число.
   - Суммарный процент всех категорий не превышает 100%.
7. Система сохраняет категорию в БД.
8. Система выводит сообщение: «Категория Покушать добавлена».

---

## Постусловия

- В базе данных создана запись о новой категории 
- Запись с привязкой к пользователю
  
---

## Альтернативные потоки

### 4a. Редактирование существующей категории

**Условие:** Пользователь хочет изменить процент уже созданной категории

**Поток:**
1. 4a1. Пользователь выбирает категорию из списка
2. 4a2. Пользователь меняет значение процента
3. 4a3. Переход к шагу 5 основного потока

### 5a. Удаление категории

**Условие:** Пользователь нажимает кнопку «Удалить» рядом с категорией

**Поток:**
1. 5a1. Система запрашивает подтверждение удаления
2. 5a2. Пользователь подтверждает действие
3. 5a3. Система удаляет категорию и пересчитывает общую сумму процентов
4. 5a4. Use-case завершается успешно

---

## Исключительные ситуации

### 6a. Суммарный процент превышает 100%

**Условие:** После ввода нового значения общая сумма долей стала 110%

**Поток:**
1. 6a1. Система выдает ошибку:"Распределение превысило 100%, это ты конечно молодец, но меня не обманишь, давай исправляй"
2. 6a2. Система не сохраняет изменения в БД
3. 6a3. Возврат к шагу 4

### 7b. База данных недоступна

**Условие:** На шаге 7 произошел разрыв соединения

**Поток:**
1. 7а1. Система получает Connection Error.
2. 7а2. Система показывает сообщение: «Ошибка связи с сервером. Настройки не сохранены»
3. 7а3. Use-case завершается неудачей

---

## Частота использования

- **Низкая** — обычно настраивается один раз при запуске приложения или при смене финансовой стратегии (раз в месяц)

## Критические требования

- **Производительность:** Сохранение настроек < 1 сек
- **Целостность:** Сумма процентов в БД для одного пользователя никогда не должна превышать 100%
- **Идемпотентность:** Повторное нажатие «Сохранить» не создает дублирующиеся категории

# Use-case 2: Добавление записи о доходе

**Предметная область:** Финучёт «Где мои деньги» 💸

**Первичный актор:** Пользователь

**Вторичные акторы:**
- Finance API (Бэкенд-сервис)
- Database (Хранилище транзакций и категорий)

**Цель:** Зафиксировать поступление денег и автоматически распределить их по категориям.

---

## Предусловия

- У пользователя создана хотя бы одна категория с указанием процента
- База данных доступна

---

## Основной поток (Happy Path)

1.  Пользователь нажимает «Добавить доход»
2.  Система отображает форму ввода суммы.
3.  Пользователь вводит сумму «1000» и нажимает «Применить»
4.  Система проверяет валидность суммы (число > 0).
5.  Система запрашивает из БД все категории пользователя с их процентами.
6.  Система рассчитывает суммы для распределения (например, 60% = 600 руб.).
7.  Система открывает транзакцию в БД.
8.  Система создает запись в таблице transactions (тип: INCOME, сумма: 1000)
9.  Система обновляет балансы в таблице category_balances для каждой категории
10. Система закрывает (commit) транзакцию.
11. Система выводит сообщение: «1000 руб. успешно распределено».

---

## Постусловия

- Общий баланс пользователя увеличен в БД
- Балансы категорий изменены в БД

---

## Альтернативные потоки

### 3a. Добавление комментария к доходу

**Условие:** Пользователь хочет пометить, за что пришли денежки

**Поток:**
1. 3a1. Пользователь вводит текст в поле «Источник» (например, «Зарплата»)
2. 3a2. Переход к шагу 4 основного потока

### 5a. Категории не настроены

**Условие:** На шаге 5 система обнаруживает, что у пользователя нет категорий.

**Поток:**
1. 5a1. Система создает транзакцию дохода.
2. 5a2. Система зачисляет всю сумму в системную категорию «Нераспределенное»
3. 5a3. Система выводит сообщение: «Доход сохранен в Нераспределенное, так как категории не настроены»
4. 5a4. Use-case завершается успешно.

---

## Исключительные ситуации

### 4a. Введено не число или отрицательное число

**Условие:**  Пользователь ввел «-500» или «abc»

**Поток:**
1. 4a1. Система подсвечивает поле ввода красным.
2. 4a2.  Система показывает сообщение: «Введите корректную положительную сумму».
3. 4a3. Возврат к шагу 3.

### 9a. Сбой при обновлении баланса категории (Atomicity Error)

**Условие:** Запись дохода создана, но при обновлении баланса категории произошла ошибка

**Поток:**
1. 9a1. Система ловит исключение БД
2. 9a2. Система выполняет ROLLBACK (откатывает создание записи о доходе).
3. 9a3. Система логирует ошибку транзакции
4. 9a4. Система показывает сообщение: «Ошибка при распределении средств. Подход, ещё подход»
5. 9a5. Use-case завершается неудачей.
---

## Частота использования

- **Средняя** — 2–4 раза в месяц (зарплата, аванс, подработки).

## Критические требования

- **Атомарность**: Доход должен либо распределиться по всем категориям, либо не распределиться вовсе.
- **Точность**: Расчет долей должен выполняться до копеек без потери остатка.

# Use-case 3: Добавление записи о доходе

**Предметная область:** Финучёт «Где мои деньги» 💸

**Первичный актор:** Пользователь

**Вторичные акторы:**
- Finance API (Бэкенд-сервис)
- Database (Хранилище транзакций и категорий)

**Цель:** Списать средства из выбранной категории.

---

## Предусловия

- У пользователя создана хотя бы одна категория с указанием процента
- База данных доступна

---

## Основной поток (Happy Path)

1. Пользователь выбирает категорию (например, «Покушать»).
2. Пользователь нажимает «Добавить расход».
3. Пользователь вводит сумму «20» и комментарий «Обэд».
4. Система проверяет, что сумма > 0.
5. Система проверяет текущий баланс категории «Покушать» в БД.
6. Система списывает 20 руб. с баланса категории.
7. Система создает запись в истории транзакций (тип: EXPENSE).
8. Система обновляет баланс на экране.

---

## Постусловия

- Общий баланс пользователя уменьшен в БД
- Балансы категорий изменены в БД

---

## Альтернативные потоки

### 1a. Быстрый расход с главного экрана

**Условие:** Пользователь хочет пометить, на что ушли денежки

**Поток:**
1. 1a1. Пользователь нажимает «Расход» без предварительного выбора категории
2. 1a2. Система предлагает выбрать категорию из выпадающего списка
3. 1a3. Переход к шагу 3 основного потока.
   
### 3a. Пользователь вводит комментарий

**Условие:** Пользователь хочет пометить, на что ушли денежки

**Поток:**
1. 3a1. Пользователь заполняет необязательное поле «Комментарий»
2. 3a2. Система сохраняет комментарий вместе с транзакцией

---

## Исключительные ситуации

### 5a. Недостаточно средств в категории

**Условие:**  На балансе категории 100 руб., а пользователь вводит расход 200.

**Поток:**
1. 5a1. Система выводит предупреждение: «В категории Питание недостаточно средств. Ты уйдёшь в минус, Брат. Продолжить?»
2. 5a2.  Пользователь нажимает «Отмена»
3. 5a3. Система закрывает форму, изменения не сохраняются.
4. 5a4. Use-case завершается успешно (без изменений).

### 7a. Ошибка конкурентного доступа

**Условие:** Пользователь одновременно внес расход с телефона и планшета

**Поток:**
1. 7a1. Система обнаруживает, что данные о балансе изменились другим процессом
2. 7a2. Система возвращает ошибку «Данные устарели»
3. 7a3. Система предлагает обновить экран и повторить ввод
4. 7a4. Use-case завершается неудачей

---

## Частота использования

- **Высокая** — 5–10 раз в день.
  
## Критические требования

- **Производительность**: Регистрация расхода < 0.5 сек.
- **Безопасность**: Пользователь может видеть и тратить только свои категории.

---

## Открытые вопросы

1. Нужно ли округлять проценты, если пользователь вводит дробные значения (например, 33.33%)?
2. Как обрабатывать возврат средств (Refund)? Это обратный расход или доход?
3. Нужно ли уведомлять пользователя, если баланс категории опустился ниже критического уровня (например, меньше 5%)?
4. Должна ли система поддерживать мультивалютность при распределении дохода?
5. Нужно ли хранить историю изменений процентных ставок в категориях для ретроспективных отчетов?
