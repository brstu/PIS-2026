class HabitController:
    """
    REST-контроллер для работы с привычкой "Пить воду".
    """

    def __init__(self, habit_service):
        self.habit_service = habit_service

    def log_water(self, request):

        command = request.to_command()

#### Исходящий адаптер: InMemoryWaterRepository

## InMemoryWaterRepository

```python
class InMemoryWaterRepository:
    def __init__(self):
        self.data = {}

    def save_entry(self, user_id: str, amount_ml: int):
        self.data.setdefault(user_id, []).append(amount_ml)

    def get_entries_for_today(self, user_id: str):
        return self.data.get(user_id, [])
```
## Принцип работы:
Данные хранятся в оперативной памяти в виде словаря.
В реальной системе этот адаптер будет заменён на PostgreSQL/MongoDB.

### Часть 6. Dependency Injection (Конфигурация зависимостей)

**DependencyContainer** (DI‑контейнер):

```python
# dependency_injection.py

from application.service.habit_service import HabitService
from infrastructure.adapter.out.in_memory_water_repository import InMemoryWaterRepository

class DependencyContainer:
    """
    DI-контейнер: связывает входящие и исходящие порты
    с их реализациями в инфраструктурном слое.
    """

    def __init__(self):
        # Исходящие адаптеры
        self.water_repository = InMemoryWaterRepository()
        self.reminder_service = None  # будет реализовано в следующих лабораторных

        # Application Service
        self.habit_service = HabitService(
            water_repository=self.water_repository,
            reminder_service=self.reminder_service
        )

    def get_habit_service(self):
        return self.habit_service
```
**Как работает DI:**

DI‑контейнер создаёт реализации исходящих портов (WaterRepository, ReminderService).

Эти реализации передаются в HabitService через конструктор.

Контроллеры и другие входящие адаптеры получают готовый сервис через методы контейнера.

Таким образом, application‑слой не создаёт зависимости сам, а получает их извне, что соответствует принципу Dependency Inversion.

### Что получилось хорошо

Удалось чётко разделить архитектуру на три независимых слоя:  
Domain, Application и Infrastructure.

Возникли сложности с пониманием роли входящих и исходящих портов, особенно когда у интерфейса пока только одна реализация.  
Также потребовалось время, чтобы правильно распределить ответственность между доменными сущностями и сервисами приложения.  
DI‑контейнер потребовал аккуратной настройки зависимостей, чтобы не нарушить принцип инверсии зависимостей.

### Что узнали нового

Познакомился с концепцией Hexagonal Architecture и принципами Ports & Adapters.  
Усвоил важность Dependency Inversion и того, что доменный слой должен быть полностью независим от инфраструктуры.  
Понял, как порты позволяют легко подменять адаптеры (например, InMemory → PostgreSQL).  
Разобрался, как DI упрощает тестирование и расширение системы.
