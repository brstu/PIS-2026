🧩 Habit Service — Архитектура сервиса
Habit Service — это модульная система, реализующая управление привычкой «Пить воду».
Сервис построен по принципам гексагональной архитектуры (Hexagonal Architecture / Ports & Adapters), что обеспечивает:

слабую связанность компонентов,

независимость доменной логики от инфраструктуры,

удобство тестирования,

возможность легко заменять адаптеры (БД, API, уведомления).

Архитектура разделена на три ключевых слоя:

🟩 1. Domain Layer — доменная модель
Это сердце системы, где находится бизнес‑логика.
Domain не знает о БД, HTTP, DI, фреймворках — он полностью изолирован.

В Habit Service доменные сущности:

Habit
Агрегат, представляющий привычку пользователя:

habit_id

user_id

список записей о воде (entries)

Методы:

add_entry(amount_ml) — добавляет запись о выпитом количестве воды.

WaterEntry
Сущность, описывающая одну запись:

объём воды

время записи (будет добавлено в ЛР‑3)

DailyGoal
Value Object, представляющий суточную цель (например, 2000 мл).

Принцип:  
Domain не зависит ни от одного внешнего слоя.
Все зависимости направлены внутрь — к доменной модели.

🟦 2. Application Layer — бизнес‑операции (use-cases)
Этот слой определяет что может делать система.
Он связывает доменную модель с внешним миром через порты.

Входящие порты (IN):
LogWaterUseCase — логирование воды

GetDailyProgressUseCase — получение прогресса за день

Эти интерфейсы вызывают контроллеры, CLI, скрипты — любые клиенты.

Исходящие порты (OUT):
WaterRepository — работа с хранилищем

ReminderService — отправка напоминаний

Application Layer зависит только от интерфейсов, а не от реализаций.

HabitService
Реализация входящих портов.
Он:

принимает команды от контроллеров,

вызывает доменную модель,

использует исходящие порты для сохранения данных или отправки уведомлений.

Принцип:  
Application Layer зависит от Domain, но не от Infrastructure.

🟨 3. Infrastructure Layer — адаптеры и внешние системы
Этот слой содержит реализации портов, то есть всё, что связано с внешним миром:

Входящие адаптеры:
HabitController — REST API (пока заглушка)

Исходящие адаптеры:
InMemoryWaterRepository — временное хранилище в памяти

EmailReminderService (будет позже) — отправка уведомлений

Конфигурация:
DependencyContainer — DI‑контейнер, который связывает:

сервисы,

репозитории,

контроллеры.

Принцип:  
Infrastructure зависит от Application, но не наоборот.

🔄 Потоки данных
Клиент вызывает REST‑эндпоинт → HabitController

Контроллер вызывает входящий порт → HabitService

HabitService:

создаёт доменные объекты,

вызывает исходящие порты (WaterRepository, ReminderService)

Infrastructure Layer сохраняет данные или отправляет уведомления.

🧱 Преимущества такой архитектуры
Тестируемость  
Можно тестировать доменную логику без БД и API.

Гибкость  
Можно заменить InMemoryRepository на PostgreSQL, не меняя код сервиса.

Чистота кода  
Domain остаётся чистым и независимым.

Масштабируемость  
Легко добавлять новые адаптеры: мобильное приложение, Telegram‑бот, cron‑задачи.

📌 Итог
Habit Service — это хорошо структурированный модуль, построенный по принципам гексагональной архитектуры.
Он разделён на три слоя, каждый из которых выполняет свою роль:

Domain — бизнес‑логика

Application — use-cases

Infrastructure — адаптеры и интеграции

Такой подход делает систему устойчивой, расширяемой и удобной для дальнейших лабораторных работ.
