2. Use-case описание
👉 Ссылка на файл: use-case.md
Основной сценарий: Отметка шага дорожной карты как выполненного
Первичный актор: Джуниор-разработчик (пользователь)
Цель: Зафиксировать выполнение конкретного шага в выбранной дорожной карте и получить ссылки на связанные ресурсы для следующего шага.
Предусловия:
•	Пользователь авторизован в системе
•	Пользователь уже выбрал и начал прохождение какой-либо дорожной карты
•	Дорожная карта содержит список шагов, которые нужно выполнять последовательно
Основной поток (Happy Path):
1.	Пользователь открывает свою текущую дорожную карту
2.	Система отображает список шагов, где текущий шаг выделен визуально
3.	Пользователь нажимает кнопку "Отметить как выполненный" рядом с текущим шагом
4.	Система проверяет, что шаг действительно является следующим в очереди
5.	Система изменяет статус шага на "Выполнен"
6.	Система фиксирует дату и время выполнения
7.	Система обновляет общий прогресс пользователя в данной дорожной карте
8.	Система определяет следующий шаг в дорожной карте
9.	Система отображает пользователю карточку следующего шага с прикрепленными ресурсами
10.	Пользователь видит обновленный прогресс и информацию о следующем шаге
Постусловия:
•	Шаг отмечен как выполненный в базе данных
•	Общий прогресс пользователя по карте пересчитан и сохранен
•	В историю активности добавлена запись о выполнении шага
•	Если шаг был последним, дорожная карта отмечается как завершенная
Альтернативные потоки:
3a. Если шаг подразумевает загрузку результата:
3a1. Система предлагает загрузить результат (ссылку на репозиторий, скриншот)
3a2. Пользователь прикрепляет ссылку или файл
3a3. Пользователь нажимает "Отправить на проверку"
3a4. Система сохраняет артефакт и меняет статус шага на "На проверке"
3a5. Возврат к шагу 7
4a. Если пользователь пытается отметить непоследовательный шаг:
4a1. Система определяет, что выбранный шаг не является текущим
4a2. Система не изменяет статус шага
4a3. Система показывает подсказку: "Сначала выполните текущий шаг"
4a4. Возврат к шагу 2
Исключительные ситуации:
5a. Если при сохранении статуса произошла ошибка базы данных:
5a1. Система откатывает транзакцию
5a2. Система логирует ошибку
5a3. Система уведомляет пользователя: "Не удалось сохранить прогресс. Попробуйте позже"
5a4. Use-case завершается неудачей
8a. Если следующий шаг не найден или данные карты повреждены:
8a1. Система сохраняет выполнение текущего шага
8a2. Система логирует критическую ошибку целостности данных
8a3. Пользователь видит сообщение: "Прогресс сохранен, но следующий шаг не найден"
8a4. Use-case завершается с частичным успехом
9a. Если не удалось загрузить ресурсы для следующего шага:
9a1. Система успешно обновила прогресс
9a2. При загрузке ресурсов происходит таймаут
9a3. Система логирует инцидент
9a4. Пользователь видит сообщение: "Шаг выполнен! Материалы временно недоступны"
9a5. Возврат к шагу 2

Описание потока:
•	Джуниор-разработчик открывает дорожную карту и видит текущий шаг
•	Пользователь нажимает "Отметить как выполненный" рядом с текущим шагом
•	API начинает транзакцию и проверяет, что шаг действительно следующий по порядку
•	Система отмечает шаг как выполненный (completed_at = NOW())
•	Progress Service пересчитывает общий прогресс пользователя (с 20% до 25%)
•	Achievement Service проверяет, достигнут ли новый рубеж, и выдает бейдж "Первые шаги"

•	Транзакция фиксируется, пользователь получает обновленный прогресс и материалы для изучения

Участники:
•	Джуниор-разработчик – пользователь, который проходит обучение по дорожной карте
•	Web UI – интерфейс для взаимодействия с пользователем
•	Roadmap API – основной бэкенд-сервис, управляющий дорожными картами и шагами
•	Progress Service – микросервис для расчета и обновления прогресса пользователя
•	Resources Service – сервис, предоставляющий обучающие материалы к шагам
•	Achievement Service – сервис достижений и бейджей для мотивации
•	PostgreSQL – база данных для хранения карт, шагов, прогресса и достижений

•	Пользователь получает сообщение с предупреждением, что прогресс временно не сохранился
•	Фоновый процесс позже проверит Redis и повторит попытку обновления БД
Участники:
•	Джуниор-разработчик – пользователь, отмечающий выполнение шага
•	Web UI – интерфейс для взаимодействия
•	Roadmap API – основной бэкенд-сервис
•	Progress Service – сервис расчета прогресса
•	Resources Service – сервис материалов (не участвует в ошибке)
•	PostgreSQL – основная БД (становится недоступной)
•	Redis – кэш-хранилище для временного сохранения данных при сбоях

4. Gherkin-сценарии
👉 Ссылка на файл: scenarios.feature
Feature: Прохождение дорожных карт
  Как джуниор-разработчик
  Я хочу отслеживать свой прогресс по дорожным картам
  Чтобы системно изучать технологии и видеть свой рост

  Background:
    Given пользователь авторизован в системе
    And у пользователя заполнен профиль: "Алексей Иванов", "alex@mail.ru"
    And пользователь начал прохождение дорожной карты "Frontend-разработчик"
    And в дорожной карте 20 шагов, текущий шаг №5 "Основы CSS"

  @happy-path
  Scenario: 01. Успешная отметка шага (Happy Path)
    When пользователь открывает дорожную карту "Frontend-разработчик"
    And пользователь нажимает кнопку "Отметить как выполненный" рядом с шагом "Основы CSS"
    Then система проверяет, что это следующий шаг по порядку
    And система меняет статус шага на "Выполнен"
    And система обновляет прогресс с 20% до 25%
    And система загружает следующий шаг "CSS Flexbox" с материалами
    And система проверяет достижения и выдаёт бейдж "Первые шаги"
    And пользователь видит сообщение "Шаг выполнен! Твой прогресс: 25%"
    And пользователь видит карточку следующего шага со ссылками на статьи и видео

  @validation @negative
  Scenario: 02. Ошибка: Попытка отметить непоследовательный шаг
    Given текущий шаг №5 "Основы CSS" ещё не выполнен
    When пользователь пытается отметить шаг №7 "JavaScript основы"
    Then система проверяет порядок выполнения
    And система определяет, что шаг №7 не является текущим
    And система НЕ меняет статус шага
    And система показывает сообщение: "Сначала выполните текущий шаг: Основы CSS"
    And прогресс пользователя остаётся 20%

  @resources @negative
  Scenario: 03. Ошибка: Не удалось загрузить ресурсы для следующего шага
    When пользователь отмечает текущий шаг "Основы CSS" как выполненный
    Then система успешно обновляет статус шага и прогресс (COMMIT выполнен)
    And система пытается загрузить ресурсы для следующего шага "CSS Flexbox"
    And сервис ресурсов возвращает ошибку "Resource service timeout"
    And система логирует ошибку загрузки материалов
    And прогресс пользователя сохранён (25%)
    And пользователь видит сообщение:
      | Шаг выполнен! Но материалы временно недоступны. Зайди позже. |
    And пользователь видит следующий шаг без прикреплённых ссылок

  @database @critical
  Scenario: 04. Ошибка: Сбой при сохранении прогресса
    Given сервис базы данных временно недоступен
    When пользователь нажимает "Отметить как выполненный"
    Then система начинает транзакцию и отмечает шаг
    And при попытке обновить общий прогресс происходит ошибка подключения к БД
    And система выполняет ROLLBACK транзакции
    And шаг возвращается в статус "Не выполнен"
    And система сохраняет данные в Redis как временное хранилище
    And пользователь видит сообщение:
      | ⚠️ Прогресс временно не сохранился, но мы это запомнили. Обнови страницу через минуту |
    And в логах фиксируется "Database connection failed, data saved to Redis"

  @achievements @bonus
  Scenario: 05. Достижение: Выполнение 25% карты (бейдж)
    Given пользователь выполнил 4 шага из 20 (прогресс 20%)
    When пользователь отмечает шаг №5 "Основы CSS"
    Then прогресс становится 25%
    And система проверяет условия для достижений
    And система создаёт запись о новом достижении "Первые шаги"
    And пользователь видит всплывающее уведомление:
      | 🏆 Получен бейдж "Первые шаги"! |
    And бейдж отображается в профиле пользователя

  @boundary @completion
  Scenario: 06. Граничный случай: Выполнение последнего шага
    Given пользователь выполнил 19 шагов из 20 (прогресс 95%)
    And последний шаг №20 "Финальный проект"
    When пользователь отмечает шаг №20 как выполненный
    Then система обновляет прогресс до 100%
    And система определяет, что все шаги выполнены
    And система меняет статус дорожной карты на "Завершена"
    And система выдаёт финальное достижение "Выпускник"
    And пользователь видит сообщение:
      | 🎉 Поздравляем! Ты полностью прошёл дорожную карту! |
    And пользователю предлагается выбрать следующую карту для изучения

Контрольные вопросы
Подготовка к защите:
1. Что такое транзакционная граница? Где она проходит в вашем сценарии?
Транзакционная граница — это логическая область, внутри которой все операции должны быть выполнены полностью (commit) или не выполнены вообще (rollback). В моем сценарии отметки шага дорожной карты транзакционная граница проходит от момента нажатия кнопки "Отметить как выполненный" до момента обновления прогресса и выдачи достижений. В эту транзакцию входят: проверка последовательности, отметка шага, обновление прогресса, проверка достижений. Загрузка ресурсов следующего шага находится ЗА границей транзакции, так как ее неудача не должна откатывать выполнение шага.
2. Почему операция X выбрана синхронной, а Y - асинхронной?
•	Синхронные (отметка шага, обновление прогресса, проверка достижений): потому что пользователь должен сразу видеть результат своих действий. Если прогресс не обновится мгновенно, пользователь подумает, что система не работает.
•	Асинхронные (отправка уведомлений, сохранение в Redis, фоновая синхронизация): потому что они не критичны для основного сценария. Пользователь может получить уведомление о бейдже через 5 секунд, это не влияет на его опыт использования.
3. Как обеспечить идемпотентность при повторных запросах?
Идемпотентность обеспечивается несколькими способами:
•	Проверка статуса шага перед выполнением (если шаг уже выполнен — повторно не отмечаем)
•	Использование idempotency_key, который передается с каждым запросом и проверяется в БД
•	Блокировки на уровне БД (optimistic/pessimistic lock) чтобы два запроса одновременно не обновили один шаг
•	В Redis сохраняем результат выполнения, чтобы при повторном запросе вернуть тот же ответ без изменений
4. Что произойдёт, если внешний сервис вернёт ошибку после частичного выполнения операции?
Зависит от места ошибки:
•	Если ошибка внутри транзакции (например, при обновлении прогресса) — выполняется ROLLBACK, все изменения откатываются, шаг не считается выполненным
•	Если ошибка после COMMIT (например, не загрузились ресурсы следующего шага) — транзакция уже зафиксирована, ошибка логируется, пользователь получает частичный функционал (шаг выполнен, но без материалов)
•	В исключительных случаях (как с Redis) данные сохраняются во временное хранилище и восстанавливаются позже
5. Как система обнаружит, что внешний сервис недоступен?
•	Таймауты — если сервис не отвечает в течение заданного времени (например, 5 секунд для Resources Service)
•	HTTP коды ответа — получение 5xx (500 Internal Server Error, 503 Service Unavailable)
•	Исключения — Connection refused, Socket timeout, Network unreachable
•	Circuit breaker pattern — если сервис недоступен длительное время, он помечается как "сломанный" и вызовы к нему временно прекращаются
•	Health checks — система мониторинга регулярно проверяет доступность сервисов
6. Какие данные нужно логировать для диагностики сбоев?
•	ID пользователя (user_id) — чтобы знать, кого затронула проблема
•	ID дорожной карты и шага (roadmap_id, step_id) — чтобы понимать, на каком этапе сбой
•	Временные метки (timestamp) — когда произошла ошибка
•	Тип ошибки и stack trace — что именно пошло не так
•	Данные запроса (request payload) — что пытался сделать пользователь
•	Статус транзакции (commit/rollback) — дошел ли запрос до конца
•	Результаты повторных попыток (retry count, success/failure) — пыталась ли система восстановиться
•	Внешние сервисы, которые были недоступны — для мониторинга зависимостей
